Diagnostics

Network diagnostics

Systematic network troubleshooting across IP, routing, DNS, MTU, socket state, and TCP internals — the skill works layer-by-layer so you never miss the root cause.

🌐
Layer 3 — IP & Routing
Inspect interface addresses with ip addr, trace the routing table with ip route show, and examine ARP cache with ip neigh. Pinpoints missing routes, wrong gateway, or interface misconfiguration.
ip addr ip route ARP
🔌
Layer 4 — TCP & Sockets
Map all listening and established connections with ss -tlnp. Diagnose TCP state machine issues — SYN_SENT, TIME_WAIT floods, FIN_WAIT leaks — and locate the owning process for any socket.
ss TCP states netstat
🔍
Application — DNS, curl & TLS
Resolve DNS with dig +short and host, compare stub resolver vs authoritative answers, walk TLS handshakes with curl -v, and pinpoint certificate chain failures or SNI mismatches.
dig curl -v TLS

Read-only first. Every diagnostic command the skill runs is non-destructive — it never modifies routing tables, flushes DNS caches, or resets interfaces without explicit confirmation from you.

Methodology

Step-by-step approach

The skill follows a strict layer-by-layer diagnostic order so it never skips ahead and misses a lower-level cause.

1
Check physical & link layer
ip link show — confirm interface is UP and RUNNING. ethtool eth0 — verify link speed, duplex, and auto-negotiation. Catches cable/NIC/driver issues before wasting time on IP.
2
Check IP & routing
ip addr show — verify the interface has the expected address and prefix. ip route show — confirm a default gateway exists and the correct metric wins. ip neigh — check ARP entries for the gateway.
3
Test connectivity
ping -c4 8.8.8.8 — proves raw ICMP reachability to the internet. traceroute or mtr — maps every hop and pinpoints where packets die, time out, or show asymmetric latency.
4
Verify application layer
ss -tlnp — confirm the service is bound and listening on the expected port. curl -sv — walk the full HTTP + TLS handshake. dig +short @8.8.8.8 — bypass the local resolver to isolate DNS from connectivity.
Reference

Key tools

Essential one-liners the skill knows and can run on your behalf.

$ ss -tlnp
# TCP listening sockets with PID / process name
$ ip route show
# Full routing table — default gw, metrics, interfaces
$ dig +short example.com
# Quick DNS resolution (A record)
$ netstat -tuln
# TCP/UDP listeners (fallback when ss not available)
$ ethtool eth0
# Physical link: speed, duplex, auto-neg, wake-on-LAN
$ tcpdump -i any host 10.0.0.1 -nn -c 50
# Capture packets to/from a host — no DNS reverse lookup

tcpdump note: The skill will ask for confirmation before running packet captures. On production hosts, always scope with -c <count> and a specific host/port filter to avoid flooding disk.

Failure Patterns

Common failures

Six failure classes the skill recognises by signature — each with a deterministic fix path.

🔴
DNS not resolving
ping 8.8.8.8 works but ping google.com fails. Root causes: systemd-resolved not running, /etc/resolv.conf points at wrong stub IP, or DNSSEC validation failure.
🗺️
Route missing
No default route in ip route show. Packets to off-subnet addresses get dropped silently. Caused by DHCP failure, NetworkManager race on boot, or manual route flush.
📦
MTU mismatch
Large packets are silently dropped on a path. Small pings work; curl hangs after TLS handshake. Classic VPN / VXLAN symptom. Fix: set ip link set dev eth0 mtu 1450 or clamp MSS via iptables.
👂
Port not listening
Service is running but ss -tlnp shows it bound to 127.0.0.1 not 0.0.0.0. Connection refused from remote hosts. Fix bind address in the app config, then restart.
🚫
ICMP blocked
ping fails but TCP connections succeed. Firewall rule drops ICMP echo. Not a connectivity bug — but masks real latency. The skill distinguishes ICMP-blocked vs truly unreachable.
🤝
TCP SYN drops
Connections time out instead of getting refused. SYN sent but no SYN-ACK. Causes: firewall, net.core.somaxconn overflow, or iptables DROP (vs REJECT) rule. Diagnose with tcpdump + conntrack.

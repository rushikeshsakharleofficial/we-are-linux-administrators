Diagnostics

Storage diagnostics

From "disk full" alerts to silent inode exhaustion — the skill covers the full storage diagnostic surface without ever running destructive commands first.

💽
Disk full
df -h shows 100% usage. The skill drills into which directory is the culprit with du -sh /* and handles the subtle case where df and du disagree due to deleted-but-open files.
🗂️
Inode exhaustion
df shows free space but you can't create files. df -i reveals 100% inode use. Common culprit: millions of small temp files or mail queue entries. The skill finds and quantifies them.
📦
LVM
lvs, pvs, vgs — inspect logical volumes, physical disks, and volume groups. Identifies inactive PVs, thin-pool overcommit, and missing volume group activation on boot.
🔄
RAID & mdadm
Read /proc/mdstat and run mdadm --detail to detect degraded arrays, failed member disks, resync progress, and estimated rebuild time.
🔧
Filesystem corruption
Check kernel logs for EXT4-fs error, XFS: Internal error, or BTRFS: tree checker messages. The skill reads dmesg and /var/log/kern.log to surface I/O errors before they cascade.
❤️
SMART failures
smartctl -a /dev/sda — reads SMART health status, reallocated sector count, pending sector count, and uncorrectable error count. Gives you early warning before a drive fully fails.

Safety gate. The skill never runs fsck, mkfs, or LVM resize operations without explicit user confirmation and a remount-ro or unmount step first.

Deep Dive

The df/du mismatch mystery

The single most confusing storage scenario: df says 99% full, du only accounts for 60%. Where did 39% go?

1
df shows 99% used
df -h /var — filesystem reports only 1 GiB free on a 100 GiB partition. Alerts firing, application writes failing. Urgency: high.
2
du shows only 60% — 39% is missing
du -sh /var only accounts for ~61 GiB. The filesystem thinks 99 GiB is used, but only 61 GiB of files exist on disk. The remaining 38 GiB is invisible to du.
3
lsof +L1 finds the deleted-but-open file
lsof +L1 lists files with a link count below 1 — i.e., deleted from the directory tree but still held open by a running process. Output shows /var/log/app.log (deleted) held by pid 9823, consuming 38 GiB.
4
Kill process → space returns instantly
The kernel only releases the blocks once the last file descriptor to the deleted inode is closed. Sending SIGHUP to reopen logs (or kill -9 9823 as a last resort) causes the kernel to reclaim all 38 GiB immediately — no fsck, no reboot.

Key commands

Essential one-liners — copy and run them or let the skill run them for you.

$ df -h
# Human-readable disk usage per mountpoint
$ du -sh /* 2>/dev/null | sort -rh | head -20
# Top directories by size — suppress permission errors
$ lsof +L1
# Files deleted but still held open (link count < 1)
$ lvs && pvs && vgs
# LVM: logical volumes, physical vols, volume groups
$ smartctl -a /dev/sda
# Full SMART health report for a block device
$ findmnt --verify --verbose
# Verify all mounts in /etc/fstab are consistent
Failure Patterns

Common failures

Five storage failure classes the skill recognises by symptom and resolves with a deterministic fix path.

👻
df/du mismatch — deleted open files
df reports high usage but du accounts for much less. Kernel is holding blocks for files deleted while still open by a process. lsof +L1 finds the culprit. Fix: SIGHUP or kill the process.
🧱
Inode exhaustion
df -h shows free space but touch newfile fails with No space left on device. df -i reveals 100% inodes consumed. Common culprit: millions of tiny temp files or mail queue.
📦
LVM not activated
After a reboot, a volume group is present (pvs shows the PV) but the logical volumes are inactive. Fix: vgchange -ay vg_name then mount. Root cause: missing lvm2-monitor.service on boot.
🚨
RAID degraded
/proc/mdstat shows [UU_] instead of [UUU]. One member disk failed. The skill reads mdadm --detail /dev/md0, identifies the failed disk, and walks you through replacement.
💥
Filesystem corruption
Kernel logs show EXT4-fs error or XFS: Internal error. The skill reads dmesg for I/O error patterns, identifies the block device, and recommends safe remediation steps (remount-ro before fsck).

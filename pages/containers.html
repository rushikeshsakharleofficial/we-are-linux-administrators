02 / Failure taxonomy

Common container failures

Three root-cause categories — the skill narrows to the specific leaf node before suggesting fixes.

🚫
Container won't start
Image pull failure — bad credentials, missing registry, network timeout.

Entrypoint crash — binary not found in image, wrong CMD args, missing env vars.

Resource limits — cgroup memory too low, CPU quota enforced at start, pids.max hit.
🔌
Networking issues
Port conflict — host port already bound; docker daemon fails silently.

Bridge / overlay — docker0 missing, IPAM exhausted, iptables FORWARD DROP.

DNS inside container — /etc/resolv.conf points to 127.0.0.11 but dnsmasq not running.
📂
Storage problems
Volume mount — source path missing on host, wrong mount type (bind vs volume).

Permission denied — UID mismatch between container process and host file owner.

SELinux label — missing :z or :Z suffix; AVC denial logged in audit.log.

04 / Rootless

Rootless containers

Podman rootless containers run entirely as an unprivileged user — safer, but with their own failure modes.

Why rootless? With rootless Podman, container processes run as your real UID on the host. A root escape in the container only gives the attacker your user account — not the full system. The trade-off is uid/gid mapping via newuidmap: each container user maps to a subuid range in /etc/subuid and /etc/subgid. Most mount failures in rootless mode trace back to a missing or wrong subuid range, or to a volume whose host path is owned by a UID outside that range.

Check rootless runtime info and uid mapping:
$ podman info --format json | jq '.host.remoteSocket, .store.runRoot'
$ podman unshare ls -la /var/data
# runs ls inside the user namespace so UIDs appear as root maps them
$ cat /etc/subuid | grep $USER
# e.g. rushikesh:100000:65536 — 65536 UIDs allocated
Common rootless fixes:
# Fix missing subuid entry
$ sudo usermod --add-subuids 100000-165535 $USER
$ sudo usermod --add-subgids 100000-165535 $USER
$ podman system migrate

# Fix volume permission (chown inside user ns)
$ podman unshare chown 0:0 /path/to/volume
05 / Commands

Key commands

Read-only inspection commands the skill runs — never modifies running containers.

Inspect & logs
$ docker inspect container_name
$ docker logs --tail 50 -f container_name
Enter namespace
# enter PID+mount ns of running container
$ nsenter --target PID --mount --pid /bin/bash
Live stats & Podman
$ docker stats --no-stream
$ podman pod ls
$ podman inspect --format '{{.HostConfig.Resources}}' app
Image history
$ docker history --no-trunc image_name
$ docker image inspect image_name | jq '.[].Config'
06 / cgroup investigation

cgroup resource checks

Walk the cgroup hierarchy to find which limit is being enforced against your container.

1
Check memory usage vs limit
Read memory.usage_in_bytes and compare against memory.limit_in_bytes. If usage is near limit, the OOM killer will fire — look for exit code 137.
$ cat /sys/fs/cgroup/memory/docker/CONTAINER_ID/memory.usage_in_bytes
$ cat /sys/fs/cgroup/memory/docker/CONTAINER_ID/memory.limit_in_bytes
2
Check CPU shares
cpu.shares (cgroup v1) or cpu.weight (cgroup v2) controls relative CPU scheduling priority. Default is 1024. Very low values throttle CPU access under contention.
$ cat /sys/fs/cgroup/cpu/docker/CONTAINER_ID/cpu.shares
$ cat /sys/fs/cgroup/cpu/docker/CONTAINER_ID/cpu.cfs_quota_us
3
Check blkio throttle
Block I/O throttling can make disk-heavy containers appear hung. Check device major:minor mappings and per-device read/write BPS limits.
$ cat /sys/fs/cgroup/blkio/docker/CONTAINER_ID/blkio.throttle.read_bps_device
$ cat /sys/fs/cgroup/blkio/docker/CONTAINER_ID/blkio.throttle.write_bps_device
4
Check pids.max
If a container forks aggressively (e.g. Celery workers, PHP-FPM), it may hit the pids limit and fail to spawn new processes — even though memory and CPU look fine.
$ cat /sys/fs/cgroup/pids/docker/CONTAINER_ID/pids.max
$ cat /sys/fs/cgroup/pids/docker/CONTAINER_ID/pids.current

9 migration types supported

From OS upgrades to DNS cutovers, the migration-expert skill covers the full spectrum of Linux server migration scenarios with purpose-built runbooks for each.

🖥️
Server OS Upgrade
CentOS 7 to Rocky 9, Ubuntu 20 to 22. In-place or parallel rebuild. Python EOL detection, SysVinit to systemd conversion, kernel ABI checks.
🔄
Full Server Clone / Rebuild
Physical to VM, VM to cloud. Full service inventory, config export, parallel rebuild with validation before cutover. Zero data loss gate.
🗄️
Database Migration
PostgreSQL version upgrades, MySQL to MariaDB transitions. Schema compatibility checks, replication setup, failover validation, rollback snapshots.
📦
Linux Patch Rollout
Fleet-wide kernel updates, security patches, staged rollout. Canary-first strategy with automated health checks after each wave before proceeding.
🛡️
Firewall & Rate-Limit Audit
Pre-migration network review. Catalogs all firewall rules, identifies redundant or conflicting entries, maps required ports for migrated services.
📁
SFTP Migration
Users, SSH keys, chroot jails, directory permissions. Exports and validates complete SFTP configuration before cutover. Key fingerprint verification included.
💾
Storage Migration
LVM resize, NFS move, block device change. Validates filesystem integrity before and after. fstab entries dry-run tested to prevent boot lockout.
⚙️
Service Migration
Moving workloads between servers. Dependency mapping, port conflict detection, parallel run period with traffic weight shifting and health check gates.
🌐
DNS Migration
Zone transfer, TTL drain, cutover. Verifies propagation at each DNS tier before old records expire. Automated rollback on validation failure.
Agent model

7 specialized parallel agents

The migration-expert skill dispatches purpose-built subagents in parallel. Each agent owns a distinct phase and writes structured output to the shared .migration/ memory map.

🎯
Migration Orchestrator
Coordinates all agents, enforces gate sequence, blocks cutover until all gates pass
📋
Inventory Agent
Catalogs services, configs, dependencies, users, crons, open ports, installed packages
⚠️
Risk Agent
Identifies breaking changes, version incompatibilities, deprecated packages, kernel ABI changes
🌐
Domain Migration Agent
Handles service-specific migration logic — databases, firewalls, SFTP, DNS — with domain expertise
✅
Validation Agent
Post-migration health checks: service up, ports listening, data integrity, performance baseline
🔄
Rollback Agent
Maintains rollback plan throughout migration, triggers automatic rollback on validation failure
🔍
Research Agent
Fetches current release notes, migration guides, and known issues when skill knowledge is insufficient
Shared state

Shared memory — the .migration/ directory

All agents read and write to a structured directory in the project root. This is the source of truth for the entire migration — reviewable, version-controllable, and auditable.

.migration/ — shared agent memory
.migration/ ├── 00-scenario.md # Migration goal and constraints ├── 01-inventory.md # Service catalog — written by Inventory Agent ├── 02-dependency-map.md # Service dependencies and startup order ├── 03-risk-register.md # Breaking changes list — written by Risk Agent ├── 04-agent-dispatch.md # Agent assignments and status ├── 05-runbook.md # Step-by-step runbook (47+ steps for major migrations) ├── 06-validation-matrix.md # Post-migration checks — written by Validation Agent ├── 07-rollback-plan.md # Rollback procedures — written BEFORE runbook └── 08-cutover-checklist.md # Final gate checklist — all agents must sign off
Lifecycle

Migration lifecycle — 7 phases

Every migration follows a gated lifecycle. No phase starts until the previous phase's deliverables are complete and written to .migration/.

Phase 1
Discovery & Inventory
Inventory Agent catalogs all services, configuration files, users, crons, open ports, and installed packages. Output written to 01-inventory.md and 02-dependency-map.md.
Phase 2
Risk Assessment
Risk Agent analyzes the inventory against the migration target. Breaking changes, version incompatibilities, deprecated packages, and kernel ABI changes are cataloged in 03-risk-register.md.
Phase 3
Runbook Generation
Based on inventory and risk register, the Orchestrator generates a step-by-step runbook in 05-runbook.md. Each step includes the command, expected output, and validation criteria.
Phase 4
Validation Setup
Validation Agent builds the post-migration health check matrix in 06-validation-matrix.md before migration begins. Checks cover service state, port availability, data integrity, and performance baselines.
Phase 5
Cutover Gate
All agents review 08-cutover-checklist.md. Backup confirmed, rollback plan written, risk register reviewed, maintenance window confirmed. No migration starts without all gates green.
Phase 6
Execution
Runbook executed step by step with live status updates. Each step is validated before the next begins. Safety hook guards all destructive operations. Rollback Agent monitors for failure conditions throughout.
Phase 7
Post-Migration Validation
Validation Agent runs the full matrix from 06-validation-matrix.md. Any failing check triggers the Rollback Agent. Pass report written with timestamps, command outputs, and performance comparisons.
Safety principles

Three non-negotiable rules

The migration-expert skill encodes these safety principles directly. They are not suggestions — they are enforced gates in the agent lifecycle.

🔄 Rollback-first
The rollback plan (07-rollback-plan.md) is written before the runbook is started. The Rollback Agent defines snapshot IDs, rollback window duration, and step-by-step recovery procedures. If rollback is not possible, migration does not proceed.
🔍 Internet research rule
When the Risk Agent encounters a package version, kernel change, or migration path that is outside the skill's training, the Research Agent is dispatched to fetch current release notes and official migration guides. No migrations proceed on stale or assumed knowledge.
💾 Backup gate
Cutover is blocked unless a confirmed, tested backup exists. The Orchestrator checks for snapshot, dump, or backup file evidence before writing the cutover gate green. "I think we have backups" is not sufficient — the backup must be verifiable.

Why rollback-first? On a production migration at 2 AM, the first question after a failure is "how do we get back?" Having the answer written down before the first command runs is what separates professional migrations from incidents.

Why fetch live docs? CentOS 7 to Rocky 9 is a complex path with evolving tooling. Python 2 EOL handling, SELinux policy gaps, and package naming differences change with each Rocky minor release. Stale knowledge causes silent failures.

Why hard-gate backups? Every "we had backups" post-mortem eventually reveals the backup was untested, on the same system being migrated, or expired. The gate forces proof before cutover.

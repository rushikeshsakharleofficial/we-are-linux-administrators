Plugin Architecture

How skills load, how safety hooks protect dangerous operations, how audit scripts collect evidence before any change.

Execution model

How a request flows through the plugin

Every invocation follows the same path — from user problem description, through the right expert skill, guarded by safety hooks before any command reaches the shell.

User Problem Description
/linux-admin:<skill>
Expert Skill Module 30+
Safety Hook PreToolUse
Audit Helper / Evidence Collection
Plan + Validation + Rollback
1
User Problem Description — plain English symptom, e.g. "nginx won't start after config change". No special syntax required.
2
/linux-admin:<skill> — the Claude Code slash command resolves to the plugin manifest and selects the right skill directory from skills/.
3
Expert Skill Module — the skill markdown is loaded, injecting a focused mental model, methodology, and command patterns for that problem domain.
4
Safety Hook (PreToolUse) — before any Bash command executes, hooks.json fires command_safety_guard.py. The command is classified as R/O, Class 2, or Class 3.
5
Audit Helper / Evidence Collection — read-only triage scripts gather structured snapshots: journal errors, service state, socket table, sysctl values, firewall ruleset.
6
Plan + Validation + Rollback — the skill produces a prioritized fix plan with validation steps and explicit rollback instructions before any write command is suggested.

Safety system

Safety hook deep dive

The PreToolUse hook intercepts every command before execution. Pattern matching classifies it into one of three risk classes and applies the appropriate policy.

🟢
Class R/O — Read-Only

Commands that only observe system state are allowed through instantly. Zero friction on evidence collection.

Examples: journalctl, cat, ss, ip route, df, du, systemctl status, ps aux, lsof, netstat, top, dmesg

🟡
Class 2 — Confirm Required

Disruptive but reversible commands. The hook halts execution, displays a risk summary, and waits for explicit user confirmation before proceeding.

Examples: systemctl stop|restart, firewall-cmd --add-rule, iptables -A, sysctl -w, useradd, chmod -R

🔴
Class 3 — Blocked Outright

Destructive, irreversible commands are denied entirely. The hook explains why the command was blocked and suggests a safe alternative approach.

Examples: rm -rf /, dd if=/dev/zero, mkfs.*, shred, wipefs, DROP DATABASE, truncate --size=0

Multi-platform

Agent platform adapters

The same expert mental model runs on three different agent platforms. Each adapter translates the skill system into the platform's native format.

🖥️
Claude Code (native)
plugin.json · hooks/hooks.json · bin/
First-class plugin support. Slash commands, PreToolUse safety hooks, and bin scripts added to the Bash PATH. Skills loaded directly via the plugin manifest. Full hook integration with the Claude Code tool call lifecycle.
🤖
Codex / OpenAI
codex/AGENTS.md
A prompt adapter that bridges the same mental model to the Codex/OpenAI Agents system. AGENTS.md encodes the skill routing logic, safety principles, and investigation methodology as agent instructions compatible with OpenAI's tool-calling format.
♊
Gemini
gemini/GEMINI.md · docs/tasks/
Gemini-specific adapter using GEMINI.md for system-level instructions. Standalone task files in docs/tasks/ cover common scenarios as self-contained Gemini task definitions — no plugin system required.
Skill anatomy

Inside a typical skill file

Every skill markdown follows the same structure. This consistency lets Claude immediately orient to the right methodology without ambiguity.

skills/diagnose/skill.md — structure example
# diagnose skill ## When to use General Linux issue triage — any problem description ## Mental model Observe → Hypothesize → Verify → Fix ## Read-only first Always gather evidence before suggesting changes. Never modify configuration without a confirmed root cause. Every write command must be preceded by a backup or rollback plan. ## Investigation sequence 1. Read journal errors: journalctl -xe --no-pager | tail -50 2. Check service state: systemctl status <service> 3. Check socket table: ss -tlnp 4. Check disk state: df -hT && df -i 5. Check recent changes: rpm -qa --last | head OR dpkg --get-selections 6. Identify failure domain → route to expert skill ## Distro detection Check /etc/os-release before any distro-specific command. RHEL family: dnf, firewalld, SELinux Debian family: apt, ufw, AppArmor ## Safety rules - R/O commands: run freely - Class 2 (disruptive): present plan, wait for confirmation - Class 3 (destructive): refuse and explain alternative

Overview

The Main Skill Router

diagnose is the entry point for every unknown Linux problem. Describe a symptom in natural language and the skill reads relevant logs, checks service state, identifies the failure domain, and hands off to a specialist — all without touching the filesystem.

🔍
Read-Only First
Every diagnosis begins by gathering evidence passively — journalctl, dmesg, systemctl status, ss, ps. No writes, no changes, no risk until a root cause is confirmed.
🧭
Automatic Routing
Once the failure domain is identified — boot, network, storage, kernel, auth, or a specific service — diagnose hands control to the correct expert skill with full context already loaded.
📋
Distro-Aware Commands
Automatically selects the right toolchain for Ubuntu/Debian apt vs RHEL dnf vs SUSE zypper, adjusting log paths, service managers, and package names accordingly.
⚠️
Safety Gates
Flags ambiguous or high-risk configurations before recommending any change. Validates fstab entries, confirms upstream reachability, and checks dependency chains in systemd before touching anything.
Methodology

The SRE Mental Model

Every diagnosis follows a strict four-phase loop borrowed from production incident response. No guessing, no shotgun fixes — structured evidence gathering at every step.

01
Observe
Collect all available signals without interpretation. Logs, metrics, service state, socket state, kernel ring buffer, resource utilisation. Build the full picture before forming any theory.
02
Hypothesize
Rank candidate root causes by probability given the observed evidence. Distinguish symptoms from causes. Generate a narrow list of testable hypotheses — not a broad shotgun.
03
Verify
Run targeted, read-only checks to confirm or eliminate each hypothesis. A hypothesis is only confirmed when evidence is conclusive, never just plausible. Every check is explained.
04
Fix
Only after root cause is confirmed: propose the minimal safe remediation, show the exact commands, and — when relevant — hand off to the specialist skill with full context pre-loaded.

Scope

What It Investigates

Three primary investigation axes — every diagnosis covers all three before forming a hypothesis.

📜
System Logs
Full-spectrum log review using journalctl -xe, dmesg, and service-specific files under /var/log/. Correlates timestamps across sources to build a timeline of failure.
⚙️
Service State
Inspects systemctl status, unit file dependencies, restart counts, activation timestamps, and failed dependency chains. Catches cascading failures before they mislead the investigation.
📊
Resource Pressure
Checks CPU saturation, memory pressure, OOM events, disk utilisation, inode exhaustion, and file descriptor limits. Many "service failures" are actually resource starvation in disguise.
Compatibility

Supported Distributions

Every command and log path is adapted at runtime for the detected distribution. No manual configuration required.

Ubuntu Debian RHEL CentOS Fedora SUSE openSUSE Arch Alpine

Distro detection is automatic. The skill reads /etc/os-release on first run and caches the result for the session. Package manager selection, log paths, and command flags are all adapted without any user configuration.

Note: Alpine Linux uses musl libc and OpenRC by default — some systemd-specific checks will be adapted automatically when Alpine is detected. Container-only installs (no /sbin/init) are handled gracefully.

Commands

Investigation Toolkit

The read-only commands that diagnose uses under the hood — all safe to run on production systems.

# Structured journal dump with context
journalctl -xe --no-pager -n 200
journalctl -u <service> --since "1 hour ago"
# Filter by priority: -p err
# Kernel ring buffer
dmesg -T --level=err,warn | tail -40
# -T = human-readable timestamps
# Service state + dependency tree
systemctl status <service> -l --no-pager
systemctl list-dependencies <service>
# Restart storm: check ActiveEnterTimestamp
# Resource pressure snapshot
cat /proc/pressure/{cpu,memory,io}
df -h && df -i
# fd limits: cat /proc/<pid>/limits

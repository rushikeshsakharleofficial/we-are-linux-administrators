02 / SSH key taxonomy

SSH key auth failure taxonomy

Walk this checklist in order — each step eliminates an entire class of failure before moving to the next.

1
Check authorized_keys permissions (must be 600)
SSH daemon with StrictModes (default: yes) refuses keys from world-readable or group-writable files. chmod 600 ~/.ssh/authorized_keys is the fix.
$ stat -c '%a %n' ~/.ssh/authorized_keys
# Expected: 600 /home/user/.ssh/authorized_keys
2
Check ~/.ssh directory ownership (must be 700)
The directory itself must be owned by the user and not group- or world-writable. A common mistake is chmod 775 .ssh during group-share setups.
$ stat -c '%a %U %n' ~/.ssh
# Expected: 700 username /home/username/.ssh
$ chmod 700 ~/.ssh
3
Check sshd_config (PubkeyAuthentication yes)
Verify the daemon is actually configured to accept public key auth. On RHEL/CentOS an include directive may override the default with a restrictive drop-in.
$ sshd -T | grep -i pubkey
# Should show: pubkeyauthentication yes
$ grep -r PubkeyAuthentication /etc/ssh/sshd_config /etc/ssh/sshd_config.d/ 2>/dev/null
4
Check SELinux context (restorecon -Rv ~/.ssh)
On SELinux-enforcing systems, a wrong file context blocks sshd from reading authorized_keys even when permissions are correct. AVC denials appear in /var/log/audit/audit.log.
$ ls -laZ ~/.ssh/authorized_keys
# Should show: system_u:object_r:ssh_home_t:s0
$ sudo restorecon -Rv ~/.ssh
5
Read /var/log/auth.log for the exact rejection reason
sshd logs the exact cause of rejection. Always read the server-side log — the client side shows only a generic "Permission denied" message even when the reason is permissions.
$ sudo grep "sshd" /var/log/auth.log | tail -30
$ sudo journalctl -u sshd --since "10 minutes ago" -f

04 / PAM

PAM stack

The Pluggable Authentication Module stack controls every authentication decision on a Linux system.

PAM auth order matters. Each line in /etc/pam.d/sshd is evaluated in sequence with a control flag: required (failure is fatal but stack continues), requisite (fail immediately), sufficient (success skips the rest), and optional (result ignored). A misplaced pam_faillock.so before pam_unix.so can lock users out immediately — even before the password is checked. Always run authselect current to see which profile is active before hand-editing PAM files.

/etc/pam.d/sshd (simplified)
#%PAM-1.0
auth required pam_sepermit.so
auth substack password-auth
auth include postlogin
account required pam_nologin.so
account include password-auth
password include password-auth
session required pam_selinux.so close
session required pam_loginuid.so
session required pam_selinux.so open env_params
session optional pam_keyinit.so force revoke
session include password-auth
session include postlogin
/etc/pam.d/sudo
#%PAM-1.0
auth include system-auth
account include system-auth
password include system-auth
session optional pam_keyinit.so revoke
session required pam_limits.so
session include system-auth

Warning: hand-editing PAM files can lock you out of the system. Always keep a root session open as a safety net, and use authselect on RHEL/CentOS or pam-auth-update on Debian/Ubuntu where available.

05 / SSSD & LDAP

SSSD / LDAP issues

When users from a directory service can't log in, walk these four checks to isolate the layer.

1
Check sssd.service status
SSSD crashing or in a restart loop is a top cause of all AD/LDAP users getting refused. The service must be active before any other check matters.
$ systemctl status sssd.service
$ journalctl -u sssd --since "30 min ago" | tail -40
2
Run sssctl user-show username
Shows whether the user is cached, when the cache entry expires, and whether the account is locked in the directory. Much faster than parsing raw SSSD debug logs.
$ sssctl user-show username
$ sssctl domain-status DOMAINNAME.LOCAL
3
Check /var/log/sssd/ for domain errors
Each SSSD domain has its own log file (e.g. sssd_DOMAINNAME.log). Enable debug level 3 temporarily to see Kerberos and LDAP transaction details.
$ ls -lht /var/log/sssd/
$ sudo tail -100 /var/log/sssd/sssd_DOMAIN.log | grep -i "error\|fail\|expire"
4
Verify KDC reachability
Kerberos auth fails silently if the KDC is unreachable. Check DNS resolution of the KDC hostname and UDP/TCP port 88 connectivity before blaming SSSD configuration.
$ dig _kerberos._tcp.DOMAIN.LOCAL SRV
$ nc -zv KDC_HOSTNAME 88 && echo "KDC reachable"
$ kinit username@DOMAIN.LOCAL && klist
06 / Commands

Key commands

Read-only auth diagnostic commands — no passwords accessed, no account states modified.

SSH verbose debug
# Run from CLIENT to see full negotiation
$ ssh -vvv user@host 2>&1 | grep -A2 "Trying\|offer\|AUTH\|key"
sshd effective config
# Shows merged config including Match blocks
$ sshd -T | grep -E "pubkey|authkey|strictmode|maxauth"
User lockout
# Check pam_faillock state
$ faillock --user username
# Reset lockout (requires root)
$ sudo faillock --user username --reset
User resolution
# Local vs directory user check
$ id username
$ getent passwd username
# Shows which NSS source resolved the user
2FA / TOTP status
# Check if google-authenticator module is in PAM
$ grep -r "pam_google_authenticator" /etc/pam.d/
# Check user TOTP secret file
$ ls -la ~/.google_authenticator
Audit log
$ sudo ausearch -m USER_AUTH --start recent -i | tail -20

Architecture

The Safety Hook

The linux-admin plugin installs a PreToolUse Bash hook via Claude Code's settings. Before Claude executes any shell command — regardless of which skill triggered it — this hook inspects the command string against the three-class classification table.

🔗
PreToolUse Event
Claude Code fires a PreToolUse hook event before every tool call. The hook receives the full command string and can return block to cancel execution or confirm to pause and ask the user. Read-only commands return nothing — they proceed immediately.
📋
Pattern Classification
The hook script matches the incoming command against two ordered deny lists: Class 3 (destructive) first, Class 2 (disruptive) second. Any command that clears both lists is treated as read-only and allowed through without interruption.
⚡
Zero Latency on R/O
Diagnostic commands — the vast majority of what linux-admin runs — are never blocked or delayed. The classification check is a fast in-process string match, not a round-trip. Read-only first is the fastest path through the gate.
🔒
Cannot Be Bypassed by Skills
Individual skills cannot disable or override the hook. It is installed at the Claude Code harness level, not inside the skill scripts. Even if a skill is updated or replaced, the safety gate remains active for all Bash tool calls.
Classification

Three-Class System

Every command fits into exactly one class. The classification is based on worst-case impact, not intent. A command that can be destructive is treated as destructive regardless of the arguments passed.

Class R/O
Read-Only
Passes immediately — no prompt

Diagnostic and inspection commands that cannot modify system state. These are the workhorses of the plugin — the hook sees them and steps aside.

Examples

journalctl systemctl status cat ls df -h ip addr ss -tlnp netstat dmesg top lsof ps aux nft list ruleset grep find
Class 2
Disruptive
Requires explicit "yes" confirmation

Commands that change running system state but whose effects are recoverable — a service can be restarted, a firewall rule can be removed, a sysctl can be reverted. High-risk enough to warrant a pause.

Examples

systemctl stop systemctl restart firewall-cmd --add-rule sysctl -w iptables -A mount service restart nft flush kill -9 modprobe
Class 3
Destructive
Blocked — requires evidence first

Irreversible operations. Once executed, data or configuration cannot be recovered from the system itself. The hook blocks these and asks Claude to provide evidence of root cause before the user can manually override.

Examples

rm -rf dd if=/dev/zero mkfs.* shred wipefs DROP TABLE passwd --delete userdel -r truncate -s 0 lvremove
Context

Why This Matters

AI confidence is not a reliable proxy for correctness. The safety gate exists precisely for cases where the model is certain but wrong.

Real scenario, paraphrased: During a disk-space incident, Claude misidentified a misbehaving application's log directory as the sole source of a write storm. It proposed rm -rf /var/log/app/ with high confidence. The safety hook blocked it and surfaced the request to the user along with a demand for evidence. Further investigation revealed the write storm was caused by a misconfigured log rotation policy — and the /var/log/app/ directory contained the only audit trail proving what had happened. Deleting it would have destroyed the evidence and left the root cause unresolved.

The gate does not assume Claude is incompetent. It assumes that even a highly capable model operating on incomplete context can produce a confident, coherent, and wrong recommendation — and that the cost of a false-positive block is a few extra seconds, while the cost of a false-negative allow is unrecoverable data loss.

Internals

Hook Architecture

The classification pipeline is a simple, auditable four-step decision tree. Every command goes through the same path in the same order.

1 Command arrives via PreToolUse event
The full command string — including flags and arguments — is passed to the hook script before Claude Code hands it to the shell executor. The hook runs synchronously; Claude waits for a response.
↓
2 Pattern match against Class 3 deny list
The command is matched against a curated set of regex patterns covering destructive operations: rm\s+-[^']*r, dd\s+.*of=, mkfs\., shred, wipefs, and others. If matched, execution is blocked immediately — no further checks.
↓
3 Pattern match against Class 2 confirm list
Commands that cleared the Class 3 check are now tested against the Class 2 set: systemctl (stop|restart|disable), sysctl -w, iptables -[AID], mount(?!\s+\-l), and similar. If matched, execution is paused and a confirmation prompt is shown.
↓
4 Allow or block/confirm before execution
Commands that clear both deny lists are treated as read-only. The hook returns without output and execution proceeds. The entire check adds sub-millisecond latency on the fast path.
↓
Allow (R/O)
Confirm (Class 2)
Block (Class 3)
Protection

What It Protects Against

Five distinct failure modes that have caused real outages when AI-assisted administration is used without a safety layer.

🎯
Misidentification
Claude correctly observes a symptom but traces it to the wrong file, service, or process. The model proposes a destructive action against the wrong target. The hook blocks and demands evidence before allowing the user to proceed.
💥
Cascade Errors
Fixing component A causes component B to fail — but the fix was already executed before the dependency was understood. Class 2 confirmation gives a pause to check dependency chains before committing to a state change.
♻️
Irreversible Changes
There is no undo for mkfs, shred, or wipefs. These are Class 3 by definition. Blocking them unconditionally means the only way to proceed is for the human operator to run the command manually — a deliberate friction point.
⌨️
Typos in Destructive Commands
A path typo in a recursive delete command is one of the oldest causes of accidental data loss in Linux. The hook catches the pattern before the shell expands globs and resolves symlinks — when the typo is still visible and correctable.
🧠
Stale Context
The model's confidence reflects the completeness of context it was given, not ground truth. If logs were truncated, a service was restarted between observations, or a file was already deleted, the model operates on stale data. The hook provides a checkpoint where the human can verify state is still as expected.
💡
Confident but Wrong
Language model confidence scores are not calibrated to real-world accuracy on system administration tasks. A very confident proposal is not safer than an uncertain one — it is just more likely to be acted on without scrutiny. The gate provides scrutiny regardless of model confidence.

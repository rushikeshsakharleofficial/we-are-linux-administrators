Overview

What It Covers

boot handles the full spectrum of Linux startup failures — from firmware-level POST problems through GRUB misconfiguration, initramfs failures, and early userspace crashes. Every recovery path is safety-gated.

💥
Kernel Panic
Post-update kernel panics from missing modules, bad initramfs, or incompatible driver builds. Identifies the panic string and routes to dracut rebuild or module loading fix.
🚨
Emergency Mode
Diagnoses why systemd dropped to emergency.target — bad fstab entry, missing mount point, failed required unit — and provides the minimal change to exit cleanly.
📦
initramfs Drop
Missing kernel module, broken crypttab, dracut build failure, or missing firmware blob. Reconstructs the correct dracut command for the running kernel version.
🧱
GRUB Recovery
Wrong root UUID, missing kernel entry, corrupted MBR/GPT. Walks through grub2-mkconfig, manual menuentry editing, and grub-install with the correct target device.
📁
fstab Corruption
Detects bad UUID, wrong fstype, missing noauto for removable media. Validates the corrected fstab entry with findmnt --verify before recommending any write.
🔧
dracut Rebuild
Guides a full initramfs rebuild for the current or a specific kernel version, with correct module inclusion flags for encrypted volumes, LVM, RAID, and network boot.
Taxonomy

Boot Failure Domains

Boot failures cluster into three distinct failure domains, each with its own diagnostic approach and recovery toolchain.

🦴
Firmware / GRUB
Failures before the kernel hands off to userspace.

No POST  Wrong root UUID  Missing kernel entry  Corrupted MBR  Secure Boot conflict
📦
initramfs
Failures inside the early boot environment before rootfs mount.

Missing module  Bad crypttab  dracut build fail  Missing firmware  LUKS key missing
🔄
Early Userspace
Failures after rootfs is mounted but before login prompt.

fstab errors  Missing mount point  systemd pivot fail  Required unit crash  SELinux relabel

Commands

Key Recovery Commands

The exact commands used in each recovery path, ready to copy into a rescue shell or chroot environment.

Emergency target entry

# At GRUB menu, append to kernel line:
systemd.unit=emergency.target
# Then mount root read-write:
mount -o remount,rw /
journalctl -xb -p err

GRUB config rebuild

# RHEL/Fedora (BIOS):
grub2-mkconfig -o /boot/grub2/grub.cfg
# RHEL/Fedora (UEFI):
grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
# Debian/Ubuntu:
update-grub

dracut initramfs rebuild

# Rebuild for running kernel:
dracut --force
# Rebuild for specific version:
dracut --force /boot/initramfs-$(uname -r).img $(uname -r)
# With verbose module list:
dracut --force --verbose 2>&1 | grep "module"

fstab validation

# Validate before reboot:
findmnt --verify --verbose
# Test-mount a specific entry:
mount -a --fake -v
# --fake = dry run, no actual mount

systemd-analyze boot blame

# Find slow units after boot:
systemd-analyze blame
systemd-analyze critical-chain
# SVG waterfall chart:
systemd-analyze plot > boot-chart.svg && xdg-open boot-chart.svg
Safety

Safety Protocol

Boot recovery carries the highest risk of any Linux administration task — a wrong fstab entry or a bad grub.cfg leaves the system unbootable. Every recommendation is gated.

Read-only first. All boot diagnostics start by reading logs (journalctl -xb, dmesg), service status, and configuration files. The skill never remounts, never edits fstab, and never writes a new initramfs without first confirming the root cause and presenting the exact change to you for review.

fstab validation before every write. If a fstab change is recommended, the corrected entry is always validated with findmnt --verify and mount -a --fake inside the recommendation — the skill will not suggest writing the file without confirming the entry parses correctly.

Never blindly remounts. Remounting root read-write in a rescue shell is shown explicitly as a required step when needed — not performed automatically. The skill always explains why a remount is necessary before instructing it.

✓
Confirm root cause first
No recovery command is issued until the failure domain is confirmed by log evidence.
✓
Show exact change
Every recommended edit is shown in full — diff format where applicable — before instructing the write.
✓
Provide rollback path
Every recovery step includes how to undo it: the previous GRUB entry, the backup initramfs, the original fstab line.

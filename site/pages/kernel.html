Overview

Kernel diagnostics

When the kernel crashes, it leaves evidence. The linux-admin skill reads that evidence systematically — from oops messages to full vmcore crash dumps.

💥
Kernel Panic
Interprets panic messages, identifies the faulting instruction via RIP, and reads the call trace to find the root cause driver or subsystem.
🐛
BUG() & WARN()
Distinguishes fatal BUG() assertions from advisory WARN() traces. Reads kernel source annotations to understand invariant violations.
🔒
Lockup Detection
Identifies soft lockups (CPU stuck in kernel >20s) vs hard lockups (NMI watchdog). Extracts held lock state from call traces.
💾
kdump Analysis
Reads vmcore crash dumps with the crash utility, navigates kernel data structures, and identifies the crash context without a live system.
📋
Call Trace Reading
Parses kernel stack frames top-to-bottom, identifies the triggering function, and maps frames back to kernel modules or out-of-tree drivers.
⚠️
Oops Messages
Decodes oops error codes (protection fault, null dereference, divide-by-zero), register state, and memory access addresses.
Reference

Anatomy of a kernel panic

A panic message looks overwhelming but has four distinct sections. linux-admin reads them in order.

1
Exception type + RIP (instruction pointer)
The first line names the panic type: general protection fault, NULL pointer dereference, BUG: kernel NULL pointer dereference. RIP shows the exact instruction address that faulted — this maps directly to a function via addr2line or objdump.
2
Registers dump
RAX, RBX, RCX, RDX, RSI, RDI and the segment registers at fault time. For pointer dereferences, RDI and RSI often hold the null or corrupt pointer. CR2 holds the faulting address on page faults.
3
Call trace (most important section)
Read bottom-to-top for causality (the bottom frames show what called into the crash). The top frame after RIP is where the fault occurred. Frames inside brackets [ ] are unreliable — trust the unbracketed ones. Module names in angle brackets (e.g. <scsi_mod>) indicate out-of-tree culprit.
4
Modules loaded
The final section lists all loaded kernel modules at crash time. Cross-reference with the call trace to identify out-of-tree or third-party modules involved. DKMS-built modules are common culprits after kernel upgrades.

Production readiness

kdump setup

A crash dump taken at the moment of panic is worth ten hours of speculation. Configure kdump before you need it.

Always have kdump configured in production. Without a vmcore, you are limited to dmesg ring buffer output which may already be overwritten by the time you look. kdump captures the full memory state at crash time and writes it to a reserved partition before the system reboots. The crash utility can then inspect kernel data structures, task lists, and page tables offline.

# Install kdump tools (RHEL/CentOS)
dnf install kexec-tools crash kernel-debuginfo

# Reserve crash kernel memory — add to GRUB_CMDLINE_LINUX in /etc/default/grub
crashkernel=auto # or fixed: crashkernel=256M

grub2-mkconfig -o /boot/grub2/grub.cfg
systemctl enable kdump --now
kdumpctl status
# /etc/kdump.conf — set crash dump path
path /var/crash
core_collector makedumpfile -l --message-level 1 -d 31
# -d 31 excludes zero pages, free pages — keeps dump small
# Load and analyse a vmcore dump
crash /usr/lib/debug/boot/vmlinux-$(uname -r) /var/crash/vmcore

# Inside crash interactive session:
crash> bt # backtrace of panicking task
crash> log # kernel message buffer
crash> ps # process list at crash time
crash> mod # loaded modules

Key commands

The first four commands to run after a suspected kernel event — even before opening a ticket.

dmesg -T | grep -i 'panic\|oops\|bug\|killed\|error\|warn' # timestamped kernel ring buffer — your first stop after any incident
journalctl -k --since today --no-pager # kernel messages from journald — survives reboots unlike dmesg
crash /usr/lib/debug/boot/vmlinux-$(uname -r) /var/crash/vmcore # open crash dump — requires kernel-debuginfo matching exact kernel version
kdumpctl status # verify kdump service and crash kernel are loaded before the next incident
# Decode a kernel address (needs System.map or vmlinux)
grep ffffffffc0a83f40 /proc/kallsyms
# Maps RIP address → function name without a debugger
dmesg journalctl -k kdump crash kdumpctl kallsyms addr2line objdump nm vmlinux

Triage cheatsheet
Kernel panic vs Oops
An oops is a recoverable kernel error — the kernel logs the trace and may continue. A panic is unrecoverable — the system halts or reboots. Oopses can precede panics if the kernel state becomes too corrupt.
Tainted kernel flag
Tainted: G = proprietary module loaded. OE = out-of-tree + externally built module. A tainted kernel makes upstream bug reports invalid — identify and remove the offending module before filing kernel bugs.
quick Is kdump armed?
kdumpctl status
Look for "Kdump is operational"
quick Last panic time
ls -lth /var/crash/
Sorted by mtime
quick Module taint check
cat /proc/sys/kernel/tainted
0 = clean kernel

Hardware fault disclaimer: Kernel panics with MCE (Machine Check Exception) errors, ECC memory errors logged in mcelog or rasdaemon, or panics that consistently occur under memory pressure may indicate failing RAM rather than a software fault. Run memtest86+ (bootable, minimum 2 passes) and check edac-util -s 4 for in-band ECC errors before assuming a driver or configuration problem. Chasing a software root cause on faulty hardware wastes hours.

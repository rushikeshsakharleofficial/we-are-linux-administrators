Overview

Performance diagnostics

From CPU saturation to I/O contention, the linux-admin skill applies a read-only-first, layered diagnostic approach — no guessing, no reboots without cause.

⚡
CPU Saturation
Distinguishes user vs sys vs iowait vs steal time. Identifies runqueue depth, scheduler latency, and throttled cgroups.
💾
Memory Pressure
Reads /proc/meminfo, tracks page faults, identifies cache eviction pressure and swap usage patterns under load.
💀
OOM Kills
Surfaces OOM kill events from dmesg and journalctl, pinpoints culprit processes, and tunes vm.overcommit_memory safely.
⏳
I/O Wait
Correlates high iowait with specific devices using iostat -x, identifies saturated block devices and offending processes.
📊
Load Average Analysis
Interprets 1m/5m/15m load averages relative to CPU count, distinguishing CPU-bound from I/O-bound saturation.
🔀
Scheduling Issues
Detects runqueue starvation, real-time priority inversions, and NUMA imbalances causing intermittent latency spikes.
Interpretation guide

Reading load average

Three numbers, one picture of system health over time. Most engineers misread them because they skip the CPU count context.

Load average is a queue length, not a percentage. Running uptime shows three numbers: 1-minute, 5-minute, and 15-minute averages. A value of 1.0 per CPU core means the system is exactly at capacity — every runnable task gets a core immediately. Values above your core count mean tasks are waiting in queue. A value of 4.0 on a 4-core system (load = 1.0/core) is fine. A value of 4.0 on a 2-core system (load = 2.0/core) means half your runnable tasks are waiting. Always divide by nproc before panicking. A rising 1-minute vs a falling 15-minute average signals an active problem; the inverse means things are recovering.

1
Check load vs core count
Run uptime and nproc --all. Divide load average by core count. Values over 1.5x sustained warrant investigation.
2
Identify trend direction
1m > 15m = problem is getting worse. 1m < 15m = recovering. Flat high across all three = sustained saturation for >15 minutes.
3
CPU-bound vs I/O-bound
High load + high %iowait in vmstat = I/O bound. High load + near-zero iowait = CPU bound. Very different remediation paths.
4
Correlate with application events
Check deploy timestamps, cron jobs, or backup windows. Load spikes rarely happen in isolation from application activity.

Key diagnostic tools

Every snippet the skill reaches for — with the flags that actually matter in production.

top -b -n1 | head -20 # single batch pass — safe for scripts
vmstat 1 5 # 5 samples, 1s interval — r=runqueue, b=blocked, wa=iowait
iostat -x 1 3 # extended stats: %util, await, svctm per device
sar -u 1 5 # CPU breakdown over 5 intervals — needs sysstat
perf top --sort comm,dso # live profiling — identifies hot functions without ptrace
strace -p $PID -c -f -T # syscall summary on running process (use with care on prod)
lsof -p $PID | wc -l # file descriptor count — high = possible fd leak or I/O storm
top vmstat iostat sar perf strace lsof iotop pidstat mpstat
Quick reference
CPU saturation signals
High load, near-zero iowait, high %user or %sys in vmstat. Runqueue depth >1.5x cores. Check with mpstat -P ALL 1 to see per-core breakdown.
I/O saturation signals
High load, high iowait, b column in vmstat >0. iostat shows device at >80% util. Processes in D (uninterruptible) state in ps aux.
quick Top CPU processes
ps aux --sort=-%cpu | head -10
quick Top memory processes
ps aux --sort=-%mem | head -10
quick I/O by process
iotop -b -n1 -o
Memory

OOM investigation

Out-of-memory kills are violent but leave evidence. Follow this sequence before touching vm.overcommit settings.

1
Check dmesg for OOM kill events
dmesg -T | grep -i "oom\|killed process\|out of memory" — timestamps tell you exactly when and how often.
2
Find which process was killed
dmesg -T | grep "Killed process" shows PID and process name. Cross-reference with systemd journal for restart history.
3
Check /proc/meminfo and page cache
cat /proc/meminfo — look at MemAvailable (not MemFree), SwapUsed, Shmem, and Slab. High Slab with low MemAvailable = kernel memory leak candidate.
4
Tune vm.overcommit_memory carefully
sysctl vm.overcommit_memory=2 with vm.overcommit_ratio=80 prevents over-allocation but may cause mmap failures. Test in staging first.

Swap usage warning: Any sustained swap activity (si or so non-zero in vmstat) means the system is memory-starved. Disk-backed swap is orders of magnitude slower than RAM. Address the root cause (memory leak, undersized instance, bloated cache) rather than adding swap space as a band-aid.

Reference

Metrics taxonomy

What each number means and where to find it — so you read context, not noise.

🧠
CPU Metrics
%user — user-space time
%sys — kernel-space time
%iowait — waiting on I/O
%steal — hypervisor stealing cycles

Source: vmstat, mpstat, sar -u
📀
Memory Metrics
MemAvailable — usable right now
Cached — page cache (reclaimable)
Buffers — block device cache
SwapUsed — spill to disk

Source: /proc/meminfo, free -h
💾
I/O Metrics
tput — MB/s throughput
IOPS — operations per second
await — avg service time (ms)
%util — device saturation (100% = full)

Source: iostat -x, iotop, blktrace

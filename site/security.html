Core philosophy

Seven Inviolable Principles

These constraints are not configurable. They are baked into every skill and enforced by the hook layer before any command runs.

🔍
Read-only first
Gather evidence before any change suggestion. No command is recommended before diagnostic data is collected and reviewed.
📋
Evidence before change
Root cause must be confirmed, not guessed. A hypothesis is not a diagnosis. Only confirmed root causes justify a change recommendation.
🚫
No blind tuning
Never suggest sysctl, ulimit, or kernel parameter changes without supporting evidence from observed system behaviour.
⚠️
No destructive command without confirmation
Class 2 commands always halt for an explicit user confirmation before the plugin proceeds. No exceptions.
💾
Backup before migration
No cutover is recommended without a confirmed backup. Migration plans that skip the backup step are rejected outright.
🔄
Rollback before cutover
A rollback plan must exist before execution starts. The plugin generates a rollback procedure as part of every migration runbook.
✅
Validate after every change
Post-change verification is not optional. Every suggested change includes explicit validation steps to confirm the fix worked.
Command classification

Three-Class Command Gate

Every Bash command is classified before execution. The classification determines whether it passes, halts for confirmation, or is blocked entirely.

Class R/O — Read-Only
Pass through instantly
Zero friction. Diagnostic commands run immediately with no prompt. These are the backbone of the evidence-gathering phase.
cat journalctl systemctl status ss ip df du ps top lsof netstat dmesg id getent
Class 2 — Disruptive
Confirm required
Command halts, a warning is shown, and the user must type 'yes' to proceed. These commands change system state but can be recovered from.
systemctl stop systemctl restart firewall-cmd --add-rule sysctl -w mount iptables -A service restart
Class 3 — Destructive
Blocked entirely
Plugin refuses outright, explains exactly why the command is blocked, and suggests a safer alternative that achieves the intended goal.
rm -rf / dd if=/dev/zero mkfs.* shred wipefs DROP TABLE passwd --delete userdel -r
Confirmation gate

Operations That Always Ask

Any operation in this list triggers a hard stop and an explicit confirmation prompt. No exceptions, no shortcuts, no context overrides.

package upgrades service restarts firewall changes fstab edits mount / remount fsck / xfs_repair sysctl writes chmod / chown recursive user deletion DB restore migration cutover
Real scenario

The Hook That Saved /var/log

A real incident where the safety gate prevented an irreversible data loss event.

At 14:32, Claude identified /var/log/app.log as a zombie file causing 38 GB of hidden disk usage. Without the safety hook, the next suggestion would have been rm -rf /var/log/. The PreToolUse hook blocked it and demanded the evidence first. The correct fix — HUP the log daemon — recovered the space without deleting anything.

$ lsof +L1 | grep app.log
# app 18432 root 4u REG 8,1 38.2G 0 /var/log/app.log (deleted)
$ kill -HUP 18432
# File descriptor released — 38 GB reclaimed
Hook architecture

How the Safety Gate Works

Four deterministic steps execute before every Bash command. The gate is synchronous and cannot be bypassed by prompt injection.

1
PreToolUse event fires before every Bash command
Claude Code's hook layer intercepts every tool call of type Bash before it reaches the shell. The command string is extracted and passed downstream.
2
scripts/command_safety_guard.py classifies the command
The guard script tests the command string against ordered pattern lists: Class 3 (destructive) checked first, then Class 2 (disruptive). Unmatched commands pass as Class R/O.
3
Class 3 match — deny with explanation and safe alternative
The hook returns a deny decision immediately. The response includes the reason for blocking and at least one safer command that achieves the user's stated intent without the destructive side-effect.
4
Class 2 match — halt and display confirmation prompt
Execution is suspended. The user sees a warning explaining exactly what the command will change, why it is classified as disruptive, and what the impact of proceeding will be. Confirmation is required before the hook allows execution.
